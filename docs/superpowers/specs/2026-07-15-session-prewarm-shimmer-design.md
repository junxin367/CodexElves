# 会话预热流光动效设计

## 目标

在 Codex 左侧会话列表中，仅对当前正在执行预热的会话标题显示蓝色流光。动效用于反馈后台预热状态，不改变预热调度、并发和 Owner 管理逻辑。

## 状态范围

- 等待队列中的会话不显示动效。
- worker 开始执行单条预热任务时显示动效。
- 单条任务成功、失败、转为前台或被运行时中断后移除动效。
- 当前并发为 4，因此最多同时显示 4 条流光。
- subagent、归档、临时、侧会话和运行中会话仍沿用现有过滤规则，不进入预热，也不显示动效。

## 展示方式

- 通过会话行的 `data-app-action-sidebar-thread-id` 与预热任务 `threadId` 精确匹配。
- 在 `[data-thread-title]` 等现有标题节点上增加状态属性，不修改标题文本和原始 DOM 层级。
- 使用伪元素叠加蓝色文字，并通过窄遮罩光带约 1.4 秒从左向右循环扫过。
- 原始文字始终保留，避免动画加载失败、主题切换或选中态下标题不可读。
- 动效不占用布局空间，不影响标题省略、悬停操作按钮和点击区域。
- 预热状态必须呈现移动光带，不使用整段标题静态变蓝作为降级效果。

## 生命周期与 DOM 同步

维护一组“正在执行预热”的 thread ID：

1. `runCodexSessionPrewarmTask` 真正开始执行时加入 ID 并同步侧边栏。
2. `finally` 中无条件移除 ID 并清理侧边栏状态，覆盖成功和异常路径。
3. 侧边栏 React 重渲染或会话行重新挂载时，现有扫描流程根据 ID 集合重新应用状态。
4. 注入运行时刷新时清理旧节点属性，防止旧脚本留下静态流光。
5. 用户在预热期间点击该会话时，沿用现有前台提升逻辑；直到恢复任务完成才移除动效。

## 前台会话优先加载

- 用户点击侧边栏会话时，Codex 原生会话加载不占用后台预热队列的 worker 并发槽位。
- 已经开始执行的预热任务继续完成，不取消、不抢占，也不释放其并发槽位。
- 点击事件继续记录交互时间；交互后的约 1200 毫秒内只暂停派发新的后台预热任务，避免与前台加载同时争用 App Server、磁盘和网络资源。
- 被点击会话如果已有后台预热任务，复用同一个任务 Promise 并提升为完整预热，禁止重复调用恢复接口。
- 被点击会话如果仍在等待队列中，前台原生加载优先；后台 worker 获取该任务时若会话已经成为当前会话，则直接标记为前台完成，不再恢复一次。
- App Server Manager 捕获成功后延迟 200 毫秒开始预热，不等待 Codex 前端页面完成加载。
- 启动预热时优先使用内存中已有的最近会话列表，不等待最近会话刷新完成；刷新在后台继续执行，列表为空时仍沿用现有重试逻辑。
- 流光状态先记录在预热线程集合中；对应会话行和标题 DOM 尚未挂载时不阻塞预热，前端扫描流程在 DOM 出现后补挂动效。
- 所有目标会话先通过后台优先级 `thread/read` 加载 turns 并写入会话缓存；内容预加载目标到此完成，不获取 Owner。
- 内容阶段全部结束后，仅对“完整恢复”目标执行 Owner 恢复；最近会话刷新在预热队列结束后运行，避免与首批内容和 Owner 恢复竞争 App Server。
- 分阶段记录内容 hydration、Owner 恢复和整体任务耗时，便于区分内容读取与运行时恢复瓶颈。
- AppServerRequestClient 探测必须复用执行中的 Promise，并对失败使用指数退避和同错误去重，避免启动阶段重复扫描与日志刷屏。

## 异常处理

- 找不到对应会话行时不报错；任务继续执行，后续侧边栏扫描会补挂动效。
- 标题节点不存在时不修改会话行，避免覆盖整行样式。
- DOM 节点被复用时，根据 thread ID 集合重新校正，清除不再预热的旧属性。
- 预热失败或 Promise 抛错时依靠 `finally` 清理，不允许动效残留。

## 验证

- 回归测试验证仅执行中的任务被标记，结束后清理。
- 验证等待任务不显示，最大同时标记数量与并发 4 一致。
- 验证侧边栏节点后挂载时可补充状态，节点复用后可清除旧状态。
- 验证失败路径、前台提升路径和窄遮罩动画样式。
- 验证用户点击不会占用预热 worker、不会取消执行中的任务，只会暂缓新任务派发。
- 验证最近会话刷新不会阻塞已有列表的首轮预热。
- 验证 Manager 捕获后约 200 毫秒即开始预热，且标题 DOM 后挂载时仍能补充流光状态。
- 验证全部内容 hydration 完成后才开始 Owner 恢复，内容预加载目标不会获取或释放 Owner。
- 验证 AppServerRequestClient 探测在失败时不会并发重复执行，并遵循退避时间。
- 执行 JavaScript 语法检查、`cdp_bridge` 测试、工作区检查、管理端类型检查和 `git diff --check`。
