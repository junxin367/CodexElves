# Codex 模型列表注入稳定性修复设计

## 背景

Codex 核心能够从 `model_catalog_json` 正确加载自定义模型，但 Codex App 的模型选择器还会受到前端白名单和既有 React 状态影响。当前 Codex App 启动后约 3 秒才完成 CodexElves 注入，导致模型列表状态可能先于补丁初始化。

运行时证据显示：

- Statsig 白名单补丁最终能够加入自定义模型。
- app-server 请求客户端补丁无法匹配新版 Codex 模块。
- 模型列表会在后续路由变化或菜单重建后突然出现。
- app-server 探测在刷新窗口内每约 120ms 重复失败并写入大量相同日志。

## 目标

- 自定义模型在首次打开模型选择器时稳定显示。
- 不依赖用户切换路由、重复打开菜单或等待偶然的 React 重渲染。
- app-server 探测失败时不并发执行、不持续刷相同日志。
- 保留 Statsig、消息响应、React 状态扫描和 app-server 四条兼容路径。
- 不修改模型目录生成、协议代理、供应商配置或管理器业务逻辑。

## 方案

采用双路径稳态修复。

### 模型列表刷新

1. catalog 加载完成后立即执行一次模型白名单刷新。
2. Statsig 白名单发生实际变化时，记录成功路径并请求模型状态刷新。
3. React 模型状态发生实际变化时，触发轻量的 UI 刷新信号，使现有菜单状态重新计算。
4. 刷新必须限频并设置最大持续时间，避免 MutationObserver 与 React 更新互相触发形成循环。
5. 已存在的消息响应补丁继续负责修补 `model/list` 返回值。

### app-server 探测治理

1. 同一时间只允许一个 app-server 模块探测任务执行。
2. 探测成功后停止后续探测。
3. 探测失败后使用递增退避时间，避免每次 120ms 刷新都重新扫描模块。
4. 相同失败状态只记录一次诊断；状态变化或经过较长冷却期后才允许再次记录。
5. 保留当前模块特征探测，后续可独立适配新版 app-server 内部接口，不阻塞本次稳定显示修复。

## 诊断

新增或完善以下诊断事件：

- Statsig 白名单首次成功应用及模型数量。
- React 模型状态首次成功修补。
- 消息响应模型列表成功修补。
- app-server 探测成功、首次失败和退避状态。
- 主动 UI 刷新触发及触发原因。

诊断事件应去重，不记录模型提示词、密钥或其他敏感内容。

## 验证

- 更新 `cdp_bridge` 注入脚本回归断言，覆盖单飞、退避、日志去重和主动刷新。
- 运行相关 Rust 测试。
- 运行 `cargo fmt --check`。
- 运行 `git diff --check`。
- 通过当前 Codex 调试端口检查 Statsig 模型列表和注入状态；需要安装新构建才能验证启动时首次显示的真实行为。

## 非目标

- 不重构整个注入脚本。
- 不移除现有兼容路径。
- 不改变模型能力、上下文窗口或思考等级计算。
- 不自动切换用户当前模型。
